243  // What follows is relevant for models with time/state dependent transits.
244  // Such transits, say from node A to node B, are functions f: int64->int64
245  // of the cumuls of a dimension. The user is free to implement the abstract
246  // RangeIntToIntFunction interface, but it is expected that the implementation
247  // of each method is quite fast. For performance-related reasons,
248  // StateDependentTransit keeps an additional pointer to a
249  // RangeMinMaxIndexFunction, with similar functionality to
250  // RangeIntToIntFunction, for g(x) = f(x)+x, where f is the transit from A to
251  // B. In most situations the best solutions are problem-specific, but in case
252  // of doubt the user may use the MakeStateDependentTransit function from the
253  // routing library, which works out-of-the-box, with very good running time,
254  // but memory inefficient in some situations.

256  RangeIntToIntFunction* transit; // f(x)
257  RangeMinMaxIndexFunction* transit_plus_identity; // g(x) = f(x) + x

268  // SUBTLE:
269  // The vehicle's fixed cost is skipped on purpose here, because we
270  // can afford to do so:
271  // - We don't really care about creating "strict" equivalence classes;
272  // all we care about is to:
273  // 1) compress the space of cost callbacks so that
274  // we can cache them more efficiently.
275  // 2) have a smaller IntVar domain thanks to using a "cost class var"
276  // instead of the vehicle var, so that we reduce the search space.
277  // Both of these are an incentive for *fewer* cost classes. Ignoring
278  // the fixed costs can only be good in that regard.
279  // - The fixed costs are only needed when evaluating the cost of the
280  // first arc of the route, in which case we know the vehicle, since we
281  // have the route's start node.
282 
283  // Only dimensions that have non-zero cost evaluator and a non-zero cost
284  // coefficient (in this cost class) are listed here. Since we only need
285  // their transit evaluator (the raw version that takes var index, not Node
286  // Index) and their span cost coefficient, we just store those.
287  // This is sorted by the natural operator < (and *not* by DimensionIndex).
288  struct DimensionCost {
289  int64 transit_evaluator_class;
290  int64 cost_coefficient;
291  const RoutingDimension* dimension;
292  bool operator<(const DimensionCost& cost) const {
293  if (transit_evaluator_class != cost.transit_evaluator_class) {
294  return transit_evaluator_class < cost.transit_evaluator_class;
295  }
296  return cost_coefficient < cost.cost_coefficient;
297  }
298  };
299  std::vector<DimensionCost>
300  dimension_transit_evaluator_class_and_cost_coefficient;

309  }

387  // accumulated at nodes along the routes. They represent quantities such as
388  // weights or volumes carried along the route, or distance or times.
389  // Quantities at a node are represented by "cumul" variables and the increase
390  // or decrease of quantities between nodes are represented by "transit"
391  // variables. These variables are linked as follows:

572  if (disjunctions_[disjunction].value.max_cardinality == max_cardinality) {
573  for (const int64 d_index : disjunctions_[disjunction].indices) {
574  f(d_index);
575  }
576  }

1305  // Stores dimensions for which global and local cumul optimizers may be used
1306  // in the corresponding dimensions_for_[global|local]_optimizer_ vectors.
1307  // This function also computes and stores the "offsets" for these dimensions,
1308  // used in the local/global optimizers to simplify LP computations.
1309  //
1310  // Note on the offsets computation:
1311  // The global/local cumul offsets are used by the respective optimizers to
1312  // have smaller numbers, and therefore better numerical behavior in the LP.
1313  // These offsets are used as a minimum value for the cumuls over the route (or
1314  // globally), i.e. a value we consider all cumuls to be greater or equal to.
1315  // When transits are all positive, the cumuls of every node on a route is
1316  // necessarily greater than the cumul of its start. Therefore, the local
1317  // offset for a vehicle can be set to the minimum of its start node's cumul,
1318  // and for the global optimizers, to the min start cumul over all vehicles.
1319  // However, to be able to distinguish between infeasible nodes (i.e. nodes for
1320  // which the cumul upper bound is less than the min cumul of the vehicle's
1321  // start), we set the offset to "min_start_cumul" - 1. By doing so, all
1322  // infeasible nodes described above will have bounds of [0, 0].
1323  // Example:
1324  // Start cumul bounds: [11, 20] --> offset = 11 - 1 = 10.
1325  // Two nodes with cumul bounds. Node1: [5, 10], Node2: [7, 20]
1326  // After applying the offset to the above windows, they become:
1327  // Vehicle: [1, 10]. Node1: [0, 0] (infeasible). Node2: [0, 10].
1328  //
1329  // On the other hand, when transits on a route can be negative, no assumption
1330  // can be made on the cumuls of nodes wrt the start cumuls, and the offset is
1331  // therefore set to 0.
