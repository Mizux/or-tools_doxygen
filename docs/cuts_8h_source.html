45 // Visible for testing. Returns a function f on integers such that:
46 // - f is non-decreasing.
47 // - f is super-additive: f(a) + f(b) <= f(a + b)
48 // - 1 <= f(divisor) <= max_scaling
49 // - For all x, f(x * divisor) = x * f(divisor)
50 // - For all x, f(x * divisor + remainder) = x * f(divisor)
51 //
52 // Preconditions:
53 // - 0 <= remainder < divisor.
54 // - 1 <= max_scaling.
55 //
56 // This is used in IntegerRoundingCut() and is responsible for "strengthening"
57 // the cut. Just taking f(x) = x / divisor result in the non-strengthened cut
58 // and using any function that stricly dominate this one is better.
59 //
60 // Algorithm:
61 // - We first scale by a factor t so that rhs_remainder >= divisor / 2.
62 // - Then, if use_letchford_lodi_version is true, we use the function described
63 // in "Strenghtening Chvatal-Gomory cuts and Gomory fractional cuts", Adam N.
64 // Letchfrod, Andrea Lodi.
65 // - Otherwise, we use a generalization of this which is a discretized version
66 // of the classical MIR rounding function that only take the value of the
67 // form "an_integer / max_scaling". As max_scaling goes to infinity, this
68 // converge to the real-valued MIR function.
69 //
70 // Note that for each value of max_scaling we will get a different function.
71 // And that there is no dominance relation between any of these functions. So
72 // it could be nice to try to generate a cut using different values of
73 // max_scaling.
74 std::function<IntegerValue(IntegerValue)> GetSuperAdditiveRoundingFunction(
75  bool use_letchford_lodi_version, IntegerValue rhs_remainder,
76  IntegerValue divisor, IntegerValue max_scaling);

78 // Given an upper bounded linear constraint, this function tries to transform it
79 // to a valid cut that violate the given LP solution using integer rounding.
80 // Note that the returned cut might not always violate the LP solution, in which
81 // case it can be discarded.
82 //
83 // What this does is basically take the integer division of the constraint by an
84 // integer. If the coefficients where doubles, this would be the same as scaling
85 // the constraint and then rounding. We choose the coefficient of the most
86 // fractional variable (rescaled by its coefficient) as the divisor, but there
87 // are other possible alternatives.
88 //
89 // Note that if the constraint is tight under the given lp solution, and if
90 // there is a unique variable not at one of its bounds and fractional, then we
91 // are guaranteed to generate a cut that violate the current LP solution. This
92 // should be the case for Chvatal-Gomory base constraints modulo our loss of
93 // precision while doing exact integer computations.
94 //
95 // Precondition:
96 // - We assumes that the given initial constraint is tight using the given lp
97 // values. This could be relaxed, but for now it should always be the case, so
98 // we log a message and abort if not, to ease debugging.
99 // - The IntegerVariable of the cuts are not used here. We assumes that the
100 // first three vectors are in one to one correspondance with the initial order
101 // of the variable in the cut.
102 //
103 // TODO(user): There is a bunch of heuristic involved here, and we could spend
104 // more effort tunning them. In particular, one can try many heuristics and keep
105 // the best looking cut (or more than one). This is not on the critical code
106 // path, so we can spend more effort in finding good cuts.
107 struct RoundingOptions {
108  bool use_mir = false;
109  IntegerValue max_scaling = IntegerValue(60);
110 };

131 // If the left variables in lp solution satisfies following inequality, we prove
132 // that there does not exist any knapsack cut which is violated by the solution.
133 // Let |Cmin| = smallest possible cover size.
134 // Let S = smallest (var_ub - lp_values[var]) first |Cmin| variables.
135 // Let cut lower bound = sum_(var in S)(var_ub - lp_values[var])
136 // For any cover,
137 // If cut lower bound >= 1
138 // ==> sum_(var in S)(var_ub - lp_values[var]) >= 1
139 // ==> sum_(var in cover)(var_ub - lp_values[var]) >= 1
140 // ==> The solution already satisfies cover. Since this is true for all covers,
141 // this method returns false in such cases.
142 // This method assumes that the constraint is preprocessed and has only non
143 // negative coefficients.

188 // Returns true if the cut is lifted. Lifting procedure is described below.
189 //
190 // First we decide a lifting sequence for the binary variables which are not

193 // Given the original constraint where the lifting candidate is fixed to one, we

204 // A cut generator that creates knpasack cover cuts.
205 //
206 // For a constraint of type
207 // \sum_{i=1..n}(a_i * x_i) <= b
208 // where x_i are integer variables with upper bound u_i, a cover of size k is a
209 // subset C of {1 , .. , n} such that \sum_{c \in C}(a_c * u_c) > b.
210 //
211 // A knapsack cover cut is a constraint of the form
212 // \sum_{c \in C}(u_c - x_c) >= 1
213 // which is equivalent to \sum_{c \in C}(x_c) <= \sum_{c \in C}(u_c) - 1.
214 // In other words, in a feasible solution, at least some of the variables do
215 // not take their maximum value.
216 //
217 // If all x_i are binary variables then the cover cut becomes
218 // \sum_{c \in C}(x_c) <= |C| - 1.
219 //
220 // The major difficulty for generating Knapsack cover cuts is finding a minimal
221 // cover set C that cut a given floating point solution. There are many ways to
222 // heuristically generate the cover but the following method that uses a
223 // solution of the LP relaxation of the constraint works the best.
224 //
225 // Look at a given linear relaxation solution for the integer problem x'
226 // and try to solve the following knapsack problem:
227 // Minimize \sum_{i=1..n}(z_i * (u_i - x_i')),
228 // such that \sum_{i=1..n}(a_i * u_i * z_i) > b,
229 // where z_i is a binary decision variable and x_i' are values of the variables
230 // in the given relaxation solution x'. If the objective of the optimal solution
231 // of this problem is less than 1, this algorithm does not generate any cuts.
232 // Otherwise, it adds a knapsack cover cut in the form
233 // \sum_{i=1..n}(z_i' * x_i) <= cb,
234 // where z_i' is the value of z_i in the optimal solution of the above
235 // problem and cb is the upper bound for the cut constraint. Note that the above
236 // problem can be converted into a standard kanpsack form by replacing z_i by 1
237 // - y_i. In that case the problem becomes
238 // Maximize \sum_{i=1..n}((u_i - x_i') * (y_i - 1)),
239 // such that
240 // \sum_{i=1..n}(a_i * u_i * y_i) <= \sum_{i=1..n}(a_i * u_i) - b - 1.
241 //
242 // Solving this knapsack instance would help us find the smallest cover with
243 // maximum LP violation.
244 //
245 // Cut strengthning:
246 // Let lambda = \sum_{c \in C}(a_c * u_c) - b and max_coeff = \max_{c
247 // \in C}(a_c), then cut can be strengthened as
248 // \sum_{c \in C}(u_c - x_c) >= ceil(lambda / max_coeff)
249 //
250 // For further information about knapsack cover cuts see
251 // A. Atamtürk, Cover and Pack Inequalities for (Mixed) Integer Programming
252 // Annals of Operations Research Volume 139, Issue 1 , pp 21-38, 2005.
253 // TODO(user): Implement cut lifting.
254 CutGenerator CreateKnapsackCoverCutGenerator(
255  const std::vector<LinearConstraint>& base_constraints,
256  const std::vector<IntegerVariable>& vars, Model* model);
